# Etapa 6.4.2 — toolchain Playwright/Chromium

**Data:** 2026-07-18

Após aprovar a 6.4.1, Lucas autorizou a 6.4.2. A imagem oficial Playwright
`1.61.0-noble` foi resolvida para o digest
`sha256:57b65fdc9ceabe0ef613124c7bbe2babcf9362c4d85e382fe3b03604e84b428a`.
O digest contém Node 24.16.0 e Chromium 149.0.7827.55; o package Playwright 1.61.0
foi instalado somente no build por `npm ci` e lockfile, sem download de browser.

O driver aceita apenas `version` e `probe`. O adapter resolve a tag local para o
image ID antes da run e usa a política Docker comum: rede desligada, rootfs e input
read-only, output separado, capabilities removidas, `no-new-privileges`, UID/GID
65534 e limites de CPU, memória, PIDs, tempo e output. A prova real iniciou Chromium
headless, reconciliou HTML em memória e confirmou bloqueio de egress.

A primeira prova rejeitou corretamente uma versão Chromium inferida de documentação
pública que não correspondia ao digest. Depois de medir o artefato real, o pin foi
reconciliado. Também foi observado que `TemporaryDirectory` no Windows cria ACL
incompatível com bind mount do Docker Desktop; a integração passou a usar raiz
controlada sob `.asef` com limpeza garantida.

A auditoria de reprodutibilidade normalizou `SOURCE_DATE_EPOCH`, árvore npm e tar
POSIX. O tar canônico foi idêntico (`c984c2e0…72bb`) em builds sem cache, assim como
versões e comportamento, mas o exporter Docker Desktop 28.2.2 gerou image IDs
distintos ao copiar o mesmo blob entre estágios. O resultado não é descrito como
byte-idêntico; o adapter resolve e preserva o image ID efetivamente executado.

Não houve compilação ou execução de `WebUiTestPlan`, acesso a site, `npm install`
durante run, compartilhamento de IPC com o host, screenshot, provider live, push,
tag ou release. A capability permanece `planned`.
