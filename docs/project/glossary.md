# Glossário

| Termo | Definição adotada no projeto |
|---|---|
| Adaptador | Implementação que traduz uma capacidade externa para um contrato interno estável. |
| Agente | Componente que usa um modelo para produzir análise, decisão ou artefato tipado dentro de limites definidos. |
| Alpha | Versão experimental usada para validar capacidades; não representa compromisso de estabilidade. |
| Aprovação humana | Decisão explícita necessária para prosseguir em uma ação ou transição sensível. |
| Artefato | Arquivo, relatório, log, código, resultado ou outro produto de uma execução. |
| Baseline | Resultado ou implementação de referência usada em comparação posterior. |
| Budget | Limite de tempo, chamadas, tokens, custo, arquivos, processos ou recursos. |
| Capability | Capacidade pequena e contratual, como executar testes ou coletar cobertura. |
| Conformance suite | Testes que verificam se uma implementação cumpre os invariantes de um contrato. |
| Dataset | Coleção versionada de casos usada para desenvolvimento, regressão, avaliação, segurança ou conformidade. |
| Developer preview | Avaliação controlada por usuários externos antes da primeira versão pública robusta. |
| Evidência | Registro verificável que sustenta uma conclusão sobre a execução. |
| Gate | Revisão formal que autoriza, ajusta, repete ou bloqueia a próxima etapa. |
| Golden test | Comparação com uma saída estrutural estável e previamente aprovada. |
| Ground truth | Referência aceita para julgar um resultado em determinado caso. |
| Holdout | Parte protegida do dataset que não participa do desenvolvimento normal. |
| LanguageProfile | Composição de adaptadores que representa um toolchain suportado de uma linguagem. |
| LLM-as-a-judge | Uso de um modelo para avaliar uma saída; nunca será evidência única quando houver verificação objetiva. |
| MCP | Protocolo para interoperabilidade de contexto e ferramentas externas; não é o mecanismo interno obrigatório de skills. |
| Milestone | Conjunto de entregáveis e critérios que demonstra uma evolução verificável. |
| Modo demo | Execução reproduzida com respostas gravadas, sem chamada real ao modelo. |
| Modo live | Execução que chama um provider real usando configuração e credencial do usuário. |
| Mutation testing | Técnica que altera sistematicamente o código para avaliar se os testes detectam os defeitos introduzidos. |
| Oracle | Mecanismo ou referência usada para determinar o resultado esperado. |
| Provider | Serviço ou runtime que disponibiliza um modelo. |
| Quality Engineering | Aplicação sistêmica de práticas de qualidade ao ciclo de engenharia, incluindo prevenção, testes, evidência, risco e melhoria. |
| Runtime | Infraestrutura que executa workflows, aplica transições, budgets, políticas, persistência e eventos. |
| Sandbox | Fronteira externa de isolamento usada para executar artefatos não confiáveis com privilégios e recursos limitados. |
| Skill | Capacidade reutilizável com contrato explícito e políticas de uso. |
| Structured output | Saída de modelo validada contra um schema conhecido. |
| SUT | System Under Test; sistema, componente ou função cuja qualidade está sendo avaliada. |
| Teste metamórfico | Teste baseado em relações esperadas entre múltiplas execuções quando um oracle completo é difícil. |
| Tool | Operação concreta que pode ser invocada por código determinístico ou agente. |
| Workflow | Sequência versionada de estados, atividades, decisões e transições para resolver um caso de uso. |
| Walking skeleton | Primeira implementação mínima que atravessa o fluxo completo e produz evidência. |

Termos novos deverão ser adicionados aqui quando adquirirem significado específico no projeto.
